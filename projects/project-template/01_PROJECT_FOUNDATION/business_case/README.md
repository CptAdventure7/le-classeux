# Business Case

Store the decision-oriented case for doing the project here in one Markdown artifact.

## Purpose

When this folder is populated, it should contain exactly one maintained artifact:
- `business_case.md`

That file should make it easy for a decision-maker to understand:
- what problem or opportunity justifies the work
- why acting now makes sense
- what value is expected and for whom
- what investment, tradeoffs, and risks come with it
- what decision, approval, or commitment is being requested

Use this folder for:
- the business or mission rationale for the project
- the recommended course of action and why it wins over alternatives
- expected value, outcomes, or avoided losses
- key cost, resource, dependency, and risk statements
- the concrete approval or next-step ask

Do not use this folder for:
- detailed scope boundaries better suited to `project_definition`
- metric catalogs better suited to `objectives_success_metrics`
- stakeholder inventories better suited to `stakeholders`
- copied roadmaps, budgets, requirements, or design detail that belong elsewhere

## Folder Rule

- Keep exactly one `.md` file in this folder: `business_case.md`.
- Create `business_case.md` only when the project has real content to capture.
- If it already exists, update it in place.
- Do not split the business case across multiple sibling files in this folder.
- Do not keep placeholder artifacts with fake content in the starter template.

## Expected Shape

`business_case.md` should stay concise, evidence-based, and decision-ready. Recommended sections:
- `Summary`
- `Problem Or Opportunity`
- `Why Now`
- `Recommended Approach`
- `Expected Value`
- `Cost And Resource View`
- `Key Risks And Dependencies`
- `Decision Or Ask`
- `Related Links`

Optional sections:
- `Alternatives Considered`
- `Assumptions`
- `Evidence`

## Authoring Rules

- Write for approval and prioritization, not for implementation detail.
- Lead with the recommendation and the reason it matters.
- Prefer quantified value, cost, and risk statements; if numbers are uncertain, say so explicitly.
- Separate evidence, assumptions, and opinion so the reader can judge confidence.
- Link to supporting requirements, plans, budget, research, or decisions instead of duplicating them.
- Remove filler and framework boilerplate that does not help a reader decide whether the project is worth doing.
- Update linked objectives, execution plans, budgets, or decisions when the business case changes materially.
- Move superseded material to `99_ARCHIVE` rather than deleting traceability.
